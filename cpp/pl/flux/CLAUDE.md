# Flux 迭代规则

这份文档把 `cpp/pl/flux` 当前已经验证过的开发流程、规范和约束沉淀成默认规则。后续每一轮迭代都应以这里为准，避免出现“代码改了，但语义漂移、性能结论不可复现、examples 和文档不同步”的情况。

## 1. 总原则

- 每次迭代都必须同时考虑正确性、语义一致性、性能和可维护性，不能只优化单一维度。
- 任何优化都不能以悄悄改变用户可观察行为为代价，尤其不能破坏逻辑表、多表流、时间边界语义和 examples 的既有行为。
- 任何“支持了某能力”或“性能提升了”的结论，都必须能被测试、examples 或 benchmark 复现，而不是只停留在描述层面。

## 2. 标准迭代流程

每次开发默认按下面顺序执行：

1. 明确本轮目标。
   目标必须落在这三类之一：正确性修复、性能优化、能力扩展。描述时要写清用户可观察结果，而不是只写实现细节。
2. 先确认影响面。
   变更是否会影响逻辑表语义、时间语义、examples、benchmark 口径、支持矩阵或 README 叙述。
3. 先补验证，再改实现。
   对 bug 修复先补回归测试；对性能问题先补 benchmark case 或至少确认已有 benchmark 能覆盖热点。
4. 实施代码修改。
   改动时优先保持现有行为边界清晰，不做无约束的大面积“顺手重构”。
5. 运行最小验证。
   先跑受影响模块的单测或 example 验证，尽早发现语义回退。
6. 运行完整验证。
   默认需要跑 `bazel test //cpp/pl/flux:all`，除非本轮只有纯文档修改且明确说明未重跑测试。
7. 同步文档。
   代码行为、benchmark 口径、支持矩阵、examples 约束、优化过程记录，必须在同一轮里更新。

## 3. 正确性规则

### 3.1 逻辑表优先

- Flux 语义默认是多逻辑表流，而不是单一扁平行数组。
- 涉及表变换的 builtin，必须优先考虑 `TableValue.tables` 语义，不能默认把多表压扁成一个结果表。
- `map`、`filter`、`limit`、`tail`、`keep`、`drop`、`rename`、`duplicate`、`set`、`reduce`、`aggregateWindow` 这类算子一旦修改，必须显式检查是否仍然保留逻辑表边界。

### 3.2 时间语义不能退回字符串近似

- 涉及 RFC3339 时间比较的逻辑，必须按解析后的时间语义处理，不能退回简单字符串比较。
- `range()` 的 `stop` 必须保持 stop-exclusive 语义。
- 任何涉及时区偏移、duration、窗口边界的修改，都必须补回归测试。

### 3.3 先修回归，再扩能力

- 如果某块语义已经被证明存在回归风险，优先先补回归测试，再做扩能力或性能优化。
- 不允许在没有锁定旧 bug 的前提下继续叠加新逻辑，否则很容易把旧问题重新带回来。

## 4. 性能规则

### 4.1 没有 benchmark，不宣称性能优化

- 任何性能结论都必须对应可重复的 benchmark、局部测试或至少一组可复验的测量结果。
- 如果只是代码层面的“看起来更快”，但没有实测数据，只能称为“可能降低开销”，不能写成“性能已提升”。

### 4.2 优先级顺序

性能优化默认按下面顺序推进，而不是一开始就做微优化：

1. 先修错误的数据组织方式。
   例如不该扁平化却扁平化、不该跨表聚合却跨表聚合。
2. 再减少重复扫描和重复构造。
   例如一次扫描同时产出 `_group` 和 group key，而不是做两遍。
3. 再替换明显的线性查找热点。
   例如把重复 `std::find` 改成预建集合。
4. 再减少字符串拼接和对象复制。
   特别关注 `pivot`、`group`、`join`、`aggregateWindow` 这类热点路径。
5. 最后才做微观层面的局部优化。

### 4.3 benchmark 结果的解释约束

- benchmark 默认是同机、同环境、同口径的前后对比工具，不是跨机器绝对性能承诺。
- 如果测量明显受到热缓存影响，必须在文档里写清楚，不得把不稳定数字包装成绝对结论。

## 5. C++ 实现规则

### 5.1 语言和工具链边界

- `cpp/pl/flux` 默认以 C++20 为语言边界开发，新增实现必须兼容当前 Bazel 配置里的 `-std=c++20`。
- 不要在这个模块里引入依赖 C++20/23 才成立的语言特性或标准库接口，除非先明确升级工具链并同步文档与构建配置。
- 新代码优先复用当前模块已在使用的基础设施，例如 `absl::Status`、`absl::StatusOr`、`std::variant`、`std::optional`、`std::shared_ptr`，不要无约束地引入第二套同类机制。

### 5.2 抽象层次

- parser、runtime value model、expression evaluation、statement execution、builtin implementation 这几层的职责必须保持清晰，不能把上层语义偷偷下沉到无关层里。
- 新增能力时优先扩已有抽象，不要为了一个 builtin 或一个 case 临时拼接旁路逻辑。
- 如果一个 helper 只服务单一 builtin，就应尽量局部化；如果多个 builtin 共享完全相同的语义或数据访问模式，再上提成公共抽象。
- 公共抽象必须以“减少重复且不损失语义清晰度”为前提；如果抽象本身让控制流更难读、状态更隐蔽，就不要为复用而复用。

### 5.3 低冗余要求

- 不允许复制粘贴大段近似实现，再靠少量分支维持差异；优先抽公共路径或统一数据流。
- 但也不要过度抽象成层层回调或模板技巧，导致调试困难。`flux` 这类执行器更适合直接、可跟踪、容易插桩的实现。
- 修改已有热点路径时，优先检查是否存在重复扫描、重复 key 构造、重复对象复制、重复字符串序列化，而不是只盯局部算法本身。

### 5.4 错误处理和可诊断性

- runtime、parser、exec 层默认使用显式错误返回，不要用隐式失败、静默吞错或仅靠注释表达“不应该发生”。
- 对用户可观察的错误，优先给出能定位问题语义的错误信息，而不是笼统的 “invalid argument”。
- 如果某个分支理论上不该发生，也应给出明确的失败路径或断言意图，避免把错误变成未定义行为。

### 5.5 内存和对象生命周期

- `Value`、`ObjectValue`、`TableValue` 相关路径的改动，必须有意识地控制拷贝次数、共享方式和对象生命周期。
- 涉及行变换、聚合、pivot、join、group 等热点时，优先避免“每行都构造完整新对象”和“按单元格重复复制整行”的实现。
- 对只读遍历，优先传 `const&`；对确实需要转移所有权的数据，显式 `std::move`，不要制造含糊的复制路径。

## 6. 执行器专属设计规则

### 6.1 语义优先于技巧

- `flux` 是脚本解析和执行器，不是通用容器算法练习场。实现上优先追求语义正确、边界清晰、便于验证，而不是炫技巧。
- 对复杂 builtin，允许写得稍微展开一些，只要能明显提升语义可读性和后续调试效率。

### 6.2 可组合但不隐式串改语义

- 每个 builtin 应尽量做到输入输出契约稳定，可与其他算子组合。
- 不能为了局部方便，在中间阶段偷偷改写 `_group`、`_time`、`_value`、`range_start`、`range_stop` 之类关键语义字段，除非这是该算子的明确职责。

### 6.3 热路径优先“数据布局正确”

- 对执行器来说，很多性能问题先是数据布局问题，再才是算法问题。
- 任何涉及 table stream、window bucket、join key、pivot key 的实现，都优先先判断数据组织是否正确，再决定要不要做更细的微优化。

### 6.4 复杂逻辑要能被单独验证

- 新增复杂 helper 或重构热点路径时，要尽量让逻辑能被局部测试覆盖，而不是只能靠端到端现象间接验证。
- 如果一个改动很难写测试，通常意味着抽象边界还不够清晰，需要先整理结构。

## 7. Benchmark 规则

- 当发现新的热点路径时，要优先判断现有 benchmark 是否覆盖；没有覆盖时，应补新的 benchmark case。
- 新增 benchmark case 时，至少要同步以下内容：
  - 数据生成逻辑
  - 对应查询模板
  - `benchmark/README.md` 中的覆盖范围说明
  - 必要时在 `benchmark/OPTIMIZATION_LOG.md` 记录引入原因和观察结论
- benchmark 的输入数据和查询应尽量代表真实热点，而不是只为得到好看的数字。

## 8. Examples 规则

- `cpp/pl/flux/examples` 下的 examples 属于行为契约的一部分，不是可有可无的演示文件。
- 每次影响 runtime 行为的迭代，都必须确认现有 examples 仍然可以执行。
- 重要 example 如果承载关键语义，应优先补成带显式断言的测试，而不只是“能跑通”。
- 当前基线要求是：CLI 单测中对全部 checked-in examples 的执行验证必须保持通过。

## 9. 测试规则

### 7.1 最低验证要求

- 默认至少运行受影响模块的单测。
- 对代码行为改动，默认还要运行 `bazel test //cpp/pl/flux:all`。
- 只有纯文档修改可以不重跑测试，但最终说明里必须明确写出“本轮未重跑测试，因为只改文档”。

### 7.2 必须补测试的场景

- 修 bug 时
- 修改逻辑表相关语义时
- 修改 `range`、时间解析、duration、一元表达式时
- 修改 `aggregateWindow`、`pivot`、`group`、`join` 这类高风险路径时
- 修改 examples 目录内容或 example 预期时
- 因 benchmark 发现功能性回归时

### 7.3 测试放置约定

- 运行时表达式和值语义优先放在 `runtime/tests/runtime_eval_unit_test.cpp`
- 执行流或更高层行为放在 `runtime/tests/runtime_exec_unit_test.cpp`
- CLI/example 约束放在 `cli/tests/flux_cli_unit_test.cpp`
- 解析器行为放在 `syntax/tests/parser_unit_test.cpp`
- connector/optimizer 模块单测放在各自的 `tests/` 子目录，避免模块根目录继续堆测试文件。

## 10. 文档规则

文档职责边界默认如下：

- `README.md`
  说明模块定位、核心能力、用户如何开始，以及关键规则入口。
- `SUPPORT_MATRIX.md`
  说明“已支持/部分支持/未支持”的能力矩阵，避免写超出真实实现的乐观描述。
- `benchmark/README.md`
  说明 benchmark 覆盖范围、运行方式、结果解释口径。
- `benchmark/OPTIMIZATION_LOG.md`
  记录优化过程、关键决策、回归教训和测量观察。
- `CLAUDE.md`
  约束后续迭代应遵循的流程、校验项和提交前标准。

只要代码行为、benchmark 口径、能力边界或 examples 约束发生变化，就必须同步对应文档。

## 11. 提交前检查清单

提交前默认逐项检查：

1. 是否破坏了逻辑表边界，或把多表流悄悄压扁了
2. 是否影响了 `range()`、时间边界、时区偏移、duration 语义
3. 是否引入了超出 C++20 边界的新特性，或引入了第二套重复机制
4. 是否出现了不必要的复制粘贴、重复扫描、重复对象复制或热路径字符串构造
5. 是否补了对应测试或 benchmark
6. 是否确认 examples 仍与实现一致
7. 是否同步了 README、支持矩阵、benchmark 文档和优化记录
8. 是否完成了 `bazel test //cpp/pl/flux:all` 或明确说明为什么没跑

## 12. 执行方式

- 后续在 `cpp/pl/flux` 的开发，默认把这份文档视为长期规则，而不是一次性说明。
- 如果后续又总结出新的稳定约束，应优先补充到这里，而不是只留在对话记录里。
- 与性能优化相关、对后续工作有复用价值的过程和结论，继续沉淀到 `benchmark/OPTIMIZATION_LOG.md`。
